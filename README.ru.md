# ffresque

Поблочное спасение файлов с повреждённых носителей.

Когда на накопителе появляются сбойные сектора (деградированное зеркало RAID, умирающий диск, повреждённый пул ZFS), такие инструменты как `rsync` и `cp` зависают или аварийно завершаются на ошибках ввода-вывода. **ffresque** копирует файлы поблочно, заполняет нечитаемые блоки нулями и отслеживает статус каждого блока в базе данных SQLite. При повторном запуске — возможно, с другой копии тех же данных (например, второго диска из разбитого зеркала) — повторяются только ранее неудавшиеся блоки.

Аналог [ddrescue](https://www.gnu.org/software/ddrescue/), но работающий на уровне файлов, а не сырых дисков.

## Требования

- Python 3.8+
- Без внешних зависимостей (только stdlib)

## Быстрый старт

```bash
# 1. Скопировать все файлы с повреждённого источника
python3 ffresque.py copy \
  --src /mnt/damaged \
  --work-dir /mnt/recovery/incomplete \
  --dst /mnt/recovery/recovered

# 2. Или указать список конкретных файлов для восстановления
python3 ffresque.py copy \
  --src /mnt/damaged \
  --work-dir /mnt/recovery/incomplete \
  --dst /mnt/recovery/recovered \
  --bad-files bad-files.txt

# 3. Проверить прогресс
python3 ffresque.py status --db blocks.db

# 4. Повторный запуск с другого источника (повторяются только плохие блоки)
python3 ffresque.py copy \
  --src /mnt/damaged-disk-b \
  --work-dir /mnt/recovery/incomplete \
  --dst /mnt/recovery/recovered \
  --db blocks.db
```

## Как это работает

```
src (повреждённый носитель) ──чтение──▶ work-dir (неполные файлы) ──перемещение──▶ dst (готовые файлы)
                                            │
                                            ▼
                                       blocks.db (статус блоков: ok / bad)
```

1. Каждый файл из `--bad-files` читается порциями по `--block-size` (по умолчанию 128K).
2. Читаемые блоки записываются в `--work-dir` с сохранением структуры каталогов. Нечитаемые блоки заполняются нулями.
3. Статус блоков (`ok` или `bad`) сохраняется в базе данных SQLite (`--db`).
4. Когда все блоки файла становятся `ok`, файл перемещается из `--work-dir` в `--dst`.
5. При повторных запусках с той же `--db` повторяются только `bad`-блоки. Это позволяет восстанавливать данные с другой копии (например, со второго диска зеркала).
6. Пути полностью восстановленных файлов дописываются в `--done-file`.
7. Файлы, уже существующие в `--dst`, пропускаются (см. `--skip-existing`).

## Команды

### `copy`

```
python3 ffresque.py copy \
  --src SRC \
  --work-dir WORK_DIR \
  --dst DST \
  [--bad-files bad-files.txt] \
  [--block-size 131072] \
  [--db blocks.db] \
  [--done-file done-files.txt] \
  [--skip-existing | --no-skip-existing] \
  [--skip-bad-blocks | --no-skip-bad-blocks]
```

| Аргумент | Обязательный | По умолчанию | Описание |
|---|---|---|---|
| `--src` | да | | Каталог-источник на повреждённом носителе |
| `--work-dir` | да | | Рабочий каталог для неполных файлов; плохие блоки заполняются нулями |
| `--dst` | да | | Каталог назначения; файлы перемещаются сюда после полного восстановления |
| `--bad-files` | нет | сканировать `--src` | Текстовый файл со списком файлов для восстановления (пути относительно `--src`, по одному на строку). Если не указан, обрабатываются все файлы из `--src` |
| `--block-size` | нет | 131072 | Размер блока чтения в байтах (128K, должен соответствовать размеру записи ФС). Не менять при повторном запуске — удалите базу и неполные файлы, начните заново |
| `--db` | нет | `blocks.db` | SQLite база для отслеживания блоков |
| `--done-file` | нет | `done-files.txt` | Файл для записи путей полностью восстановленных файлов |
| `--skip-existing` | нет | включён | Пропускать файлы, уже существующие в `--dst`; отключить: `--no-skip-existing` |
| `--skip-bad-blocks` | нет | выключен | Пропускать файлы, у которых все блоки уже были опробованы, даже если среди них есть плохие; включить: `--skip-bad-blocks` |

#### `--skip-existing`

Включён по умолчанию. Если файл уже существует в `--dst`, он считается полностью восстановленным и пропускается. Отключите через `--no-skip-existing` для принудительной переобработки (например, если файл в `--dst` был повреждён после восстановления).

#### `--skip-bad-blocks`

Выключен по умолчанию. При включении файлы, у которых каждый блок уже был прочитан хотя бы раз (независимо от того, остались ли `bad`-блоки), пропускаются. Полезно при повторном запуске с **того же** источника — постоянно нечитаемые блоки не будут перечитываться. При переключении на **другой** источник (например, другой диск зеркала) оставьте этот флаг выключенным, чтобы плохие блоки были повторно прочитаны с нового носителя.

### `status`

```
python3 ffresque.py status [--db blocks.db]
```

Выводит статистику восстановления из базы данных: общее число файлов, завершённые/повреждённые, топ-10 наиболее повреждённых файлов и общий процент восстановления.

## Восстановление из нескольких источников

Основной сценарий повторных запусков: разбитое зеркало, где каждый диск имеет разные сбойные сектора.

```bash
# Источник A — первый проход
python3 ffresque.py copy \
  --src /mnt/disk-a \
  --work-dir /mnt/recovery/incomplete \
  --dst /mnt/recovery/recovered \
  --bad-files bad.txt \
  --db blocks.db

# Источник B — повторяются только блоки, не прочитанные с диска A
python3 ffresque.py copy \
  --src /mnt/disk-b \
  --work-dir /mnt/recovery/incomplete \
  --dst /mnt/recovery/recovered \
  --bad-files bad.txt \
  --db blocks.db
```

После обоих запусков файлы, у которых два диска имели плохие блоки на разных смещениях, будут полностью восстановлены.

## Сценарии применения

- **Умирающий HDD с плохими секторами** — скопировать фото, документы и видео с отказывающего жёсткого диска, пока он не умер окончательно. В отличие от `cp` и `rsync`, ffresque не зависает на нечитаемых секторах.
- **Деградированное зеркало RAID1 / ZFS** — один диск мёртв, у выжившего рассеяны сбойные сектора. Запустите ffresque на диске A, затем подключите диск B и запустите повторно — сбойные сектора на разных смещениях компенсируются данными с другого диска.
- **Пул ZFS / BTRFS с ошибками данных** — `zpool status` или `btrfs scrub` показывают ошибки контрольных сумм. Смонтируйте пул readonly и спасайте файлы поблочно.
- **Восстановление данных с NAS** — старый Synology / QNAP / TrueNAS с деградированным массивом. Извлеките диски, подключите к Linux-машине и спасите что возможно.
- **Восстановление с образа диска** — работа с `dd`-образами повреждённых дисков через loopback-монтирование. ffresque прозрачно обрабатывает ошибки ввода-вывода из образа.
- **SD-карта / CF-карта / USB-флешка** — повреждённая карта памяти камеры или флешка с нечитаемыми секторами. Восстановить максимум фотографий.
- **rsync завершается с ошибкой I/O** — `rsync: read errors mapping: Input/output error`. ffresque пропускает плохие блоки вместо прерывания всего файла.
- **Частичное восстановление файлов** — даже если некоторые блоки потеряны безвозвратно, остальная часть файла сохраняется. Для медиафайлов (JPEG, MP4) это часто означает, что большая часть содержимого остаётся пригодной.
- **Объединение нескольких резервных копий** — у вас есть несколько старых бэкапов одного дерева каталогов, каждый с разными повреждениями. Запустите ffresque поочерёдно с каждого источника, чтобы собрать наиболее полную версию.
- **Спасение перед миграцией** — перед выводом из эксплуатации старого сервера или массива скопировать всё с поблочным отслеживанием, чтобы точно знать, что было и не было восстановлено.

## Схема базы данных

```sql
-- Статус отдельных блоков
CREATE TABLE blocks (
    file TEXT NOT NULL,
    block_num INTEGER NOT NULL,
    status TEXT NOT NULL,        -- 'ok' | 'bad'
    updated_at TEXT,
    PRIMARY KEY (file, block_num)
);

-- Сводка по файлам
CREATE TABLE files (
    file TEXT PRIMARY KEY,
    size INTEGER NOT NULL,
    total_blocks INTEGER NOT NULL,
    ok_blocks INTEGER DEFAULT 0,
    bad_blocks INTEGER DEFAULT 0,
    complete BOOLEAN DEFAULT 0
);
```
